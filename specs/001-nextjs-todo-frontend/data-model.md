# Data Model

**Feature**: Next.js Todo App Frontend (Phase II)
**Date**: 2026-01-08
**Purpose**: Define frontend data structures and their relationships

## Overview

This document defines the data structures used in the Next.js frontend. Since this is a frontend-only feature, the data model represents the **client-side view** of entities, not the database schema. The backend API owns the canonical data model.

## Entities

### Task

Represents a todo item in the application.

**Source**: Backend API (GET /tasks, POST /tasks, PATCH /tasks/{id})

**TypeScript Interface**:

```typescript
// shared/types/task.ts
export type TaskPriority = 'low' | 'medium' | 'high';

export interface Task {
  id: string;                    // UUID or numeric ID from backend
  title: string;                 // Required, max 200 characters
  description: string | null;    // Optional, max 1000 characters
  completed: boolean;            // Default: false
  priority: TaskPriority;        // Default: 'low'
  tags: string[];                // Array of tag strings, can be empty
  created_at: string;            // ISO 8601 timestamp from backend
  updated_at: string;            // ISO 8601 timestamp from backend
}
```

**Zod Schema** (for runtime validation):

```typescript
// lib/schemas/task.ts
import { z } from 'zod';

export const TaskPrioritySchema = z.enum(['low', 'medium', 'high']);

export const TaskSchema = z.object({
  id: z.string(),
  title: z.string().min(1).max(200),
  description: z.string().max(1000).nullable(),
  completed: z.boolean(),
  priority: TaskPrioritySchema,
  tags: z.array(z.string()),
  created_at: z.string().datetime(),
  updated_at: z.string().datetime(),
});

export type Task = z.infer<typeof TaskSchema>;
```

**Field Descriptions**:

| Field | Type | Required | Constraints | Notes |
|-------|------|----------|-------------|-------|
| id | string | Yes | Non-empty | Generated by backend, immutable |
| title | string | Yes | 1-200 chars | Primary task description |
| description | string \| null | No | 0-1000 chars | Extended details, can be null or empty |
| completed | boolean | Yes | true/false | Task completion status |
| priority | enum | Yes | low/medium/high | Visual priority indicator |
| tags | string[] | Yes | Array | Can be empty array, no duplicates recommended |
| created_at | string | Yes | ISO 8601 | Set by backend on creation |
| updated_at | string | Yes | ISO 8601 | Updated by backend on modification |

**Validation Rules**:

1. **Title**:
   - MUST NOT be empty or only whitespace
   - MUST be ≤ 200 characters
   - Trimmed before submission

2. **Description**:
   - Can be null or empty string
   - MUST be ≤ 1000 characters if provided
   - Trimmed before submission

3. **Priority**:
   - MUST be one of: 'low', 'medium', 'high'
   - Defaults to 'low' if not specified

4. **Tags**:
   - Each tag MUST be non-empty string
   - Duplicates should be removed client-side
   - Case-sensitive
   - No maximum count (reasonable limit: 10)

5. **Timestamps**:
   - Read-only from frontend perspective
   - Validated as ISO 8601 format
   - Used for display and sorting

**State Transitions**:

```
[New Task Form]
    ↓ (submit with title)
[Optimistic Task] (temporary ID, pending: true)
    ↓ (API success)
[Persisted Task] (real ID from backend)
    ↓ (user edits)
[Updated Task] (optimistic update)
    ↓ (API success)
[Persisted Task] (updated fields)
    ↓ (user deletes)
[Removed from UI] (optimistic delete)
    ↓ (API success)
[Permanently Deleted]
```

**Error States**:

- **Optimistic Create Failure**: Remove from list, show error toast, reopen modal with data
- **Optimistic Update Failure**: Revert to previous state, show error toast, reopen modal
- **Optimistic Delete Failure**: Restore to list, show error toast

## Client-Side Only Types

### CreateTaskInput

Data structure for creating a new task (before backend assigns ID).

```typescript
// lib/types/task-input.ts
export interface CreateTaskInput {
  title: string;                 // Required
  description?: string | null;   // Optional
  priority?: TaskPriority;       // Optional, defaults to 'low'
  tags?: string[];               // Optional, defaults to []
}
```

**Zod Schema**:

```typescript
export const CreateTaskInputSchema = z.object({
  title: z.string().min(1, 'Title is required').max(200, 'Title too long'),
  description: z.string().max(1000, 'Description too long').nullable().optional(),
  priority: TaskPrioritySchema.optional(),
  tags: z.array(z.string().min(1)).optional(),
});
```

---

### UpdateTaskInput

Data structure for updating an existing task (partial updates allowed).

```typescript
// lib/types/task-input.ts
export interface UpdateTaskInput {
  title?: string;
  description?: string | null;
  completed?: boolean;
  priority?: TaskPriority;
  tags?: string[];
}
```

**Zod Schema**:

```typescript
export const UpdateTaskInputSchema = z.object({
  title: z.string().min(1).max(200).optional(),
  description: z.string().max(1000).nullable().optional(),
  completed: z.boolean().optional(),
  priority: TaskPrioritySchema.optional(),
  tags: z.array(z.string().min(1)).optional(),
}).refine(
  (data) => Object.keys(data).length > 0,
  { message: 'At least one field must be provided' }
);
```

---

### OptimisticTask

Extended Task type with optimistic update metadata.

```typescript
// lib/types/optimistic-task.ts
export interface OptimisticTask extends Task {
  _optimistic?: {
    pending: boolean;           // True during API request
    action: 'create' | 'update' | 'delete';
    previousState?: Task;       // For rollback on failure
  };
}
```

**Usage**: React Query mutations use this to track pending operations and enable rollback.

## Data Flow

### Fetching Tasks

```
[Component Mount]
    ↓
[React Query: useQuery('tasks')]
    ↓
[API: GET /tasks]
    ↓
[Validate with TaskSchema]
    ↓
[Cache in React Query]
    ↓
[Render TaskList]
```

### Creating Task

```
[User fills form]
    ↓
[Validate with CreateTaskInputSchema]
    ↓
[React Query: useMutation]
    ↓
[Optimistic Update: Add to cache with temp ID]
    ↓
[API: POST /tasks]
    ↓ (success)
[Replace temp ID with real ID]
    ↓ (failure)
[Remove from cache, show error]
```

### Updating Task

```
[User edits task]
    ↓
[Validate with UpdateTaskInputSchema]
    ↓
[React Query: useMutation]
    ↓
[Optimistic Update: Update in cache]
    ↓
[API: PATCH /tasks/{id}]
    ↓ (success)
[Confirm update]
    ↓ (failure)
[Rollback to previous state, show error]
```

### Deleting Task

```
[User clicks delete]
    ↓
[React Query: useMutation]
    ↓
[Optimistic Update: Remove from cache]
    ↓
[API: DELETE /tasks/{id}]
    ↓ (success)
[Confirm deletion]
    ↓ (failure)
[Restore to cache, show error]
```

## Caching Strategy

**React Query Configuration**:

```typescript
// lib/react-query/config.ts
export const queryConfig = {
  queries: {
    staleTime: 1000 * 60 * 5,      // 5 minutes
    cacheTime: 1000 * 60 * 30,     // 30 minutes
    refetchOnWindowFocus: true,     // Refresh on tab focus
    refetchOnReconnect: true,       // Refresh on network reconnect
    retry: 3,                       // Retry failed requests 3 times
  },
  mutations: {
    retry: 1,                       // Retry mutations once
  },
};
```

**Cache Keys**:
- `['tasks']` - All tasks list
- `['tasks', taskId]` - Individual task (future enhancement)

**Invalidation Strategy**:
- After successful create: Invalidate `['tasks']`
- After successful update: Invalidate `['tasks']`
- After successful delete: Invalidate `['tasks']`
